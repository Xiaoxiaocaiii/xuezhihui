# 1-项目详细说明书

## 一、项目目的

### 1.1 项目概述

**学知汇·大学版**是一款专为大学生群体打造的轻量级学习资源共享平台。平台以"让知识流动起来"为核心理念，致力于解决高校学生在日常学习过程中面临的学习资料分散、优质资源获取困难、同学之间缺乏高效共享渠道等实际痛点。

### 1.2 核心定位

在大学校园环境中，学生们往往面临以下问题：

- **资料分散**：课件、笔记、复习资料分散在各个同学的电脑中，缺乏集中共享的平台
- **获取困难**：优质的期末复习笔记、实验报告模板等资源往往只在少数人手中流传，无法惠及更多同学
- **交流缺失**：学生之间缺少围绕学习资料进行讨论交流的便捷渠道

学知汇正是为了解决上述问题而诞生的。平台提供了一站式的学习资源上传、浏览、下载和评论功能，构建了一个互助共享的校园学习社区。

### 1.3 核心功能范围

本平台完整实现了以下九大核心功能：

| 编号 | 功能 | 说明 |
|------|------|------|
| 1 | 用户注册 | 新用户创建账号，填写用户名和密码完成注册 |
| 2 | 用户登录 | 已注册用户通过用户名和密码登录系统 |
| 3 | 退出登录 | 已登录用户安全退出，清除本地会话信息 |
| 4 | 上传学习资源 | 用户上传学习资料文件（支持任意格式），填写资源标题 |
| 5 | 下载学习资源 | 用户浏览资源列表并一键下载感兴趣的学习资料 |
| 6 | 展示资源上传者 | 每个资源卡片上清晰展示上传者的用户名 |
| 7 | 删除个人上传的资源 | 上传者可以删除自己发布的资源 |
| 8 | 发表资源评论 | 用户对任意资源发表评论，进行学习交流 |
| 9 | 删除个人发表的评论 | 评论者可以删除自己发表的评论 |

### 1.4 适用场景

- **期末复习**：学生上传课程笔记和复习资料，全班同学共享下载
- **实验课程**：分享实验报告模板、实验数据处理方法
- **考研备考**：共享历年真题、复习笔记、知识点总结
- **日常学习**：课后作业参考资料、课程PPT的集中管理

## 三、项目过程

### 3.1 需求分析阶段

在项目启动之初，团队首先对目标用户群体——大学生进行了需求调研和分析。

**用户痛点分析**：
1. 大学课程学习中，课堂笔记、课件资料分散在各个学生手中，缺乏统一获取渠道
2. 临近期末考试或课程设计时，学生对往年复习资料、实验模板的需求强烈
3. 现有的文件共享方式（QQ群文件、U盘拷贝）存在文件过期、查找不便等问题
4. 学生希望能够围绕具体的学习资料进行讨论和交流

**功能需求定义**：
基于上述分析，团队确定了三大核心功能模块：

```
需求拆解：
├── 用户模块
│   ├── 用户注册（用户名+密码）
│   └── 用户登录（用户名+密码验证）
├── 资源模块
│   ├── 上传资源（文件+标题）
│   ├── 浏览资源列表
│   ├── 下载资源文件
│   └── 删除个人资源（权限控制）
└── 评论模块
    ├── 发表评论（关联资源+评论内容）
    ├── 查看评论列表
    └── 删除个人评论（权限控制）
```

**非功能性需求**：
- 平台轻量化：前端无需安装依赖，打开即用
- 接口简洁：后端API遵循RESTful规范，返回统一JSON格式
- 跨平台兼容：纯Web应用，任何带浏览器的设备均可使用

### 3.3 数据库设计阶段

#### 3.3.1 实体关系分析

在需求分析的基础上，团队识别出三个核心实体及其关系：

```
┌──────────┐        ┌──────────────┐        ┌──────────┐
│   user   │ 1───N  │   resource   │ 1───N  │  comment │
│  用户表   │        │    资源表     │        │   评论表   │
└──────────┘        └──────────────┘        └──────────┘
     │                                              │
     └────────────────── 1───N ─────────────────────┘
                  (用户与评论的直接关联)
```

- **user ↔ resource**：一对多关系，一个用户可以上传多个资源
- **user ↔ comment**：一对多关系，一个用户可以发表多条评论
- **resource ↔ comment**：一对多关系，一个资源可以有多条评论

#### 3.3.2 数据表结构设计

**user（用户表）**：

```sql
CREATE TABLE user (
    id       INT          NOT NULL AUTO_INCREMENT COMMENT '用户ID',
    username VARCHAR(50)  NOT NULL                COMMENT '用户名（唯一）',
    password VARCHAR(100) NOT NULL                COMMENT '密码（明文）',
    PRIMARY KEY (id),
    UNIQUE KEY uk_username (username)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户表';
```

设计要点：
- `id`为主键，使用`AUTO_INCREMENT`自动生成用户唯一标识
- `username`设置`UNIQUE KEY`约束，确保用户名不重复
- 字段均设置`NOT NULL`，保证数据完整性

**resource（资源表）**：

```sql
CREATE TABLE resource (
    id        INT          NOT NULL AUTO_INCREMENT COMMENT '资源ID',
    title     VARCHAR(200) NOT NULL                COMMENT '资源标题',
    file_path VARCHAR(500) NOT NULL                COMMENT '服务器存储路径',
    file_name VARCHAR(200) NOT NULL                COMMENT '原始文件名',
    user_id   INT          NOT NULL                COMMENT '上传者ID',
    username  VARCHAR(50)  NOT NULL DEFAULT ''     COMMENT '上传者用户名',
    PRIMARY KEY (id),
    KEY idx_user_id (user_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='资源表';
```

设计要点：
- `file_path`存储服务器上的完整文件路径，`file_name`存储用户上传时的原始文件名
- `user_id`建立与user表的关联关系，`username`冗余存储以便查询时无需JOIN
- `idx_user_id`普通索引加速按上传者筛选资源的查询

**comment（评论表）**：

```sql
CREATE TABLE comment (
    id          INT          NOT NULL AUTO_INCREMENT COMMENT '评论ID',
    resource_id INT          NOT NULL                COMMENT '资源ID',
    user_id     INT          NOT NULL                COMMENT '评论者ID',
    username    VARCHAR(50)  NOT NULL DEFAULT ''     COMMENT '评论者用户名',
    content     TEXT         NOT NULL                COMMENT '评论内容',
    PRIMARY KEY (id),
    KEY idx_resource_id (resource_id),
    KEY idx_user_id (user_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='评论表';
```

设计要点：
- `resource_id`和`user_id`分别建立索引，加速按资源和按用户查询评论
- `content`使用`TEXT`类型，支持较长的评论内容
- `username`冗余存储评论者用户名，与resource表保持一致的设计模式

#### 3.3.3 SQL语句设计

团队为每个业务功能设计了对应的SQL语句，共9条：

| 编号 | 所属Mapper | SQL类型 | 功能 | 对应SQL |
|------|-----------|---------|------|---------|
| 1 | UserMapper | INSERT | 插入新用户 | `INSERT INTO user (username, password) VALUES (#{username}, #{password})` |
| 2 | UserMapper | SELECT | 按用户名查询 | `SELECT id, username, password FROM user WHERE username = #{username}` |
| 3 | ResourceMapper | INSERT | 插入新资源 | `INSERT INTO resource (title, file_path, file_name, user_id, username) VALUES (...)` |
| 4 | ResourceMapper | SELECT | 查询全部资源 | `SELECT ... FROM resource ORDER BY id DESC` |
| 5 | ResourceMapper | SELECT | 按ID查询资源 | `SELECT ... FROM resource WHERE id = #{id}` |
| 6 | ResourceMapper | DELETE | 按ID删除资源 | `DELETE FROM resource WHERE id = #{id}` |

#### 3.4.4 用户注册接口

```java
@PostMapping("/register")
public ResultUtil<?> register(@RequestBody User user) {
    // 第一步：校验用户名非空
    if (user.getUsername() == null || user.getUsername().trim().isEmpty()) {
        return ResultUtil.error("用户名不能为空");
    }
    // 第二步：校验密码非空
    if (user.getPassword() == null || user.getPassword().trim().isEmpty()) {
        return ResultUtil.error("密码不能为空");
    }
    // 第三步：检查用户名是否已存在（唯一性校验）
    if (userMapper.selectByUsername(user.getUsername()) != null) {
        return ResultUtil.error("用户名已存在");
    }
    // 第四步：插入新用户记录
    userMapper.insert(user);
    return ResultUtil.success("注册成功");
}
```

**业务流程**：
1. 前端发送POST请求至`/api/user/register`，请求体为`{"username":"...", "password":"..."}`
2. Spring自动将JSON反序列化为`User`对象
3. 后端逐层校验：用户名非空 → 密码非空 → 用户名唯一性
4. 校验通过后调用`userMapper.insert(user)`，MyBatis执行`INSERT INTO user (username, password) VALUES (?, ?)`
5. 返回`{code: 200, message: "注册成功", data: "注册成功"}`

```java
@PostMapping("/login")
public ResultUtil<?> login(@RequestBody User user) {
    User exist = userMapper.selectByUsername(user.getUsername());
    if (exist == null || !exist.getPassword().equals(user.getPassword())) {
        return ResultUtil.error("用户名或密码错误");

    exist.setPassword(null);  // 安全处理：清除返回给前端的密码字段

}
```

**业务流程**：
1. 前端发送POST请求至`/api/auth/login`
2. 后端按用户名查询数据库：`SELECT id, username, password FROM user WHERE username = ?`
3. 验证用户存在且密码匹配
4. 将返回对象的`password`字段置为`null`，防止密码泄露到前端
5. 返回用户信息`{id, username}`给前端，前端存入localStorage

#### 3.4.6 文件上传接口

```java
@PostMapping("/upload")
public ResultUtil<?> upload(@RequestParam("file") MultipartFile file,
                            @RequestParam("title") String title,
                            @RequestParam("userId") Integer userId,
                            @RequestParam("username") String username) throws IOException {
    // 步骤1：确保上传目录存在
    new File(uploadDir).mkdirs();
    // 步骤2：生成唯一文件名（时间戳 + 原始文件名）
    String filename = System.currentTimeMillis() + "_" + file.getOriginalFilename();
    String filePath = uploadDir + "/" + filename;
    // 步骤3：将上传的文件保存到服务器磁盘
    file.transferTo(new File(filePath));
    // 步骤4：构建Resource实体对象
    Resource resource = new Resource();
    resource.setTitle(title);
    resource.setFilePath(filePath);
    resource.setFileName(file.getOriginalFilename());
    resource.setUserId(userId);
    resource.setUsername(username);
    // 步骤5：将资源记录插入数据库
    resourceMapper.insert(resource);
    return ResultUtil.success(resource);
}
```

**文件存储原理**：
- 上传目录：`{项目根目录}/uploads/`
- 文件命名策略：`{当前毫秒时间戳}_{原始文件名}`，例如`1780993826400_高等数学笔记.pdf`
- 时间戳前缀确保文件名唯一，避免同名文件覆盖
- 文件通过`MultipartFile.transferTo()`方法写入磁盘
- 文件元信息（标题、路径、原始文件名、上传者）存入数据库

#### 3.4.7 文件下载接口

```java
@GetMapping("/{id}/download")
public ResponseEntity<byte[]> download(@PathVariable Integer id) throws IOException {
    // 步骤1：根据ID查询资源记录
    Resource resource = resourceMapper.selectById(id);
    if (resource == null) return ResponseEntity.notFound().build();
    // 步骤2：从磁盘读取文件内容为字节数组
    byte[] bytes = Files.readAllBytes(new File(resource.getFilePath()).toPath());
    // 步骤3：对中文文件名进行URL编码
    String encodedFilename = URLEncoder.encode(resource.getFileName(), "UTF-8").replace("+", "%20");
    // 步骤4：构建HTTP响应（二进制流 + 下载头）
    return ResponseEntity.ok()
            .header(HttpHeaders.CONTENT_DISPOSITION,
                    "attachment; filename=\"" + encodedFilename + "\"; filename*=UTF-8''" + encodedFilename)
            .body(bytes);
}
```

**下载实现原理**：
1. 前端发起GET请求至`/api/resources/{id}/download`
2. 后端查询资源记录获取文件路径和原始文件名
3. 使用`Files.readAllBytes()`将文件全部读入内存（适合小文件场景）
4. 设置`Content-Disposition`响应头，指定`attachment`方式下载
5. `filename*=UTF-8''`是RFC 5987标准，确保中文文件名正确显示
6. 前端通过fetch获取Blob，创建临时URL触发浏览器下载

#### 3.4.8 权限控制实现（删除资源）

```java
@DeleteMapping("/{id}")
public ResultUtil<?> delete(@PathVariable Integer id, @RequestBody Map<String, Object> body) {
    // 步骤1：查询资源是否存在
    Resource resource = resourceMapper.selectById(id);
    if (resource == null) return ResultUtil.error("资源不存在");
    // 步骤2：权限校验——提取请求中的userId，与资源上传者ID比较
    int userId = Integer.parseInt(body.get("userId").toString());
    if (!resource.getUserId().equals(userId)) return ResultUtil.error("无权删除");
    // 步骤3：执行删除
    resourceMapper.deleteById(id);
    return ResultUtil.success("删除成功");
}
```

**权限控制逻辑**：采用"属主校验"模式——前端发送请求时携带当前用户的`userId`，后端将其与数据库中资源的`user_id`字段比较，只有上传者本人才能删除自己的资源。评论删除采用相同的逻辑。

```javascript
async function request(method, url, data, isFormData) {
    var headers = {};
    if (!isFormData) headers['Content-Type'] = 'application/json';
    var body = data ? (isFormData ? data : JSON.stringify(data)) : null;
    var response = await fetch(BASE_URL + url, { method: method, headers: headers, body: body });
    var result = await response.json();
    if (result.code !== 200) throw new Error(result.message || '操作失败');
    return result;
}
```

该函数封装了：
- Content-Type自动设置（JSON请求 vs FormData文件上传）
- JSON序列化（JS对象自动转为JSON字符串）
