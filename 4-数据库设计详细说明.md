# 4-数据库设计详细说明.md

## 一、数据库整体设计说明

### 1.1 数据库基本信息

| 属性 | 值 | 说明 |
|------|-----|------|
| 数据库名称 | `xuezhihui` | 与项目名称对应 |
| 默认字符集 | `utf8mb4` | 支持完整Unicode字符集，包括emoji表情和扩展汉字 |
| 默认排序规则 | `utf8mb4_unicode_ci` | 不区分大小写的Unicode排序，`ci`=Case Insensitive |
| 存储引擎 | `InnoDB` | MySQL默认事务型存储引擎，支持行级锁、外键、崩溃恢复 |
| MySQL版本 | 8.3 | MySQL 8.x系列最新版本 |

**字符集选择说明**：
- **为什么用`utf8mb4`而非`utf8`**：MySQL中的`utf8`实际上是阉割版，最多只支持3字节UTF-8编码（无法存储emoji和部分扩展汉字），`utf8mb4`是真正的4字节UTF-8完整实现
- **排序规则**：`utf8mb4_unicode_ci`基于Unicode官方排序算法，比`utf8mb4_general_ci`更准确但稍慢。对于学习资源共享平台而言，数据量不大，准确性优先

### 1.2 数据表整体关系

本数据库包含**三张核心数据表**：

```
┌──────────────┐          ┌──────────────┐          ┌──────────────┐
│     user     │          │   resource   │          │   comment    │
│    用户表     │          │    资源表     │          │    评论表     │
├──────────────┤          ├──────────────┤          ├──────────────┤
│ id (PK)      │──┐       │ id (PK)      │──┐       │ id (PK)      │
│ username(UQ) │  │   ┌──▶│ user_id (IDX)│  │   ┌──▶│ resource_id  │
│ password     │  │   │   │ title        │  │   │   │ user_id (IDX)│
└──────────────┘  │   │   │ file_path    │  │   │   │ username     │
                  │   │   │ file_name    │  │   │   │ content      │
                  └───┤   │ username     │  └───┤   └──────────────┘
                      │   └──────────────┘      │
                      │   user_id → 外键逻辑     │   user_id → 外键逻辑
                      │   （一个用户 → 多个资源）  │   （一个用户 → 多条评论）
                      │                         │   resource_id → 外键逻辑
                      │                         │   （一个资源 → 多条评论）
                      └─────────────────────────┘
```

**表间关系说明**：

| 关系 | 类型 | 关联字段 | 说明 |
|------|------|----------|------|
| user → resource | 一对多 | `user.id` ↔ `resource.user_id` | 一个用户可以上传多个学习资源 |
| user → comment | 一对多 | `user.id` ↔ `comment.user_id` | 一个用户可以发表多条评论 |
| resource → comment | 一对多 | `resource.id` ↔ `comment.resource_id` | 一个资源可以接收多条评论 |

**设计说明**：本数据库的关联关系通过**逻辑外键**实现（即在业务代码中维护数据一致性），而非在数据库层面使用`FOREIGN KEY`约束。这是轻量级应用中常见的设计选择，避免了外键约束带来的性能开销和删除级联的复杂性。

## 二、建库建表脚本 schema.sql 逐行解析

**文件路径**：`src/main/resources/sql/schema.sql`

### 2.1 创建数据库

```sql
-- 创建数据库
CREATE DATABASE IF NOT EXISTS xuezhihui
    DEFAULT CHARACTER SET utf8mb4
    COLLATE utf8mb4_unicode_ci;
```

**逐关键字解析**：

`CREATE DATABASE` — DDL语句，创建新数据库。

`IF NOT EXISTS` — 条件判断关键字。如果数据库`xuezhihui`已经存在，则不执行创建操作，也不报错。这使得脚本可以**重复执行**（幂等性），适合开发环境和自动化部署。

`DEFAULT CHARACTER SET utf8mb4` — 设置数据库的默认字符集为`utf8mb4`。此后在该数据库中创建的表，如果不显式指定字符集，将继承此设置。`utf8mb4`每个字符最多使用4个字节存储，兼容所有Unicode字符。

`COLLATE utf8mb4_unicode_ci` — 设置数据库的默认排序规则。`utf8mb4_unicode_ci`的各部分含义：
- `utf8mb4`：字符集
- `unicode`：基于Unicode官方排序算法（UCA）
- `ci`：Case Insensitive，不区分大小写。例如`a`和`A`在比较时视为相等

```sql
USE xuezhihui;
```

**`USE xuezhihui`** — 切换当前数据库上下文。后续的`CREATE TABLE`语句将在`xuezhihui`数据库中创建表。如果缺少此语句，后续建表语句将默认创建到当前连接的默认数据库中（可能为其他数据库）。

`CREATE TABLE user` — 创建名为`user`的表。`user`在某些数据库中可能是保留字，在MySQL中虽然不是保留字，但建议用反引号包裹以避免潜在问题。此处未使用反引号，因为在MySQL 8.x中`user`直接使用是安全的。

```sql

```

**`id`字段逐属性解析**：

| 属性 | 值 | 含义 |
|------|-----|------|
| 字段名 | `id` | 用户唯一标识，主键 |
| 数据类型 | `INT` | 整数类型，占用4字节，取值范围-2,147,483,648到2,147,483,647。对于学生平台而言，INT完全足够（可存储21亿用户） |

```sql
    username VARCHAR(50)  NOT NULL                COMMENT '用户名（唯一）',
```

**`username`字段逐属性解析**：

| 属性 | 值 | 含义 |
|------|-----|------|
| 字段名 | `username` | 用户名，登录凭据之一 |
| 数据类型 | `VARCHAR(50)` | 可变长字符串，最大50个字符。`VARCHAR`只占用实际字符数+1~2字节的长度空间（相比`CHAR`固定占用50字符空间更节省存储） |
| `NOT NULL` | — | 非空约束。用户名必须填写，不允许空值 |
| `COMMENT '用户名（唯一）'` | — | 注释说明了此字段有唯一性约束（通过下方的`UNIQUE KEY`实现） |

**业务含义**：
- `username`是用户登录的凭据之一（与`password`配合）
- 用于在资源卡片和评论列表中展示用户身份（"👤 上传者：zhangsan"）
- 唯一性：同一用户名只能被注册一次，注册接口会先查询此字段判断是否重复
- 长度限制50字符，前端输入框有`maxlength="16"`限制，实际存储上限远大于前端限制

---

```sql
    password VARCHAR(100) NOT NULL                COMMENT '密码（明文）',
```

**`password`字段逐属性解析**：

| 属性 | 值 | 含义 |
|------|-----|------|
| 字段名 | `password` | 密码字段 |
| 数据类型 | `VARCHAR(100)` | 可变长字符串，最大100字符。之所以设为100（而非前端限制的16），是为了兼容未来可能的密码加密扩展（如BCrypt加密后的密文长度约60字符） |
| `NOT NULL` | — | 非空约束 |
| `COMMENT '密码（明文）'` | — | 注释明确标注了当前阶段密码以明文形式存储 |

**业务含义**：
- 当前版本使用明文存储和比较（`user.getPassword().equals(inputPassword)`）
- 登录成功后，后端将密码字段置为`null`再返回给前端，避免密码泄露
- 前端限制密码长度最短3位、最长16位
- 生产环境应使用BCrypt或类似算法加密存储

---

```sql
    PRIMARY KEY (id),
```

**`PRIMARY KEY (id)`** — 主键约束：

**主键的作用**：
1. **唯一标识**：确保`id`列的值在整个表中唯一，不重复
2. **自动索引**：MySQL自动为主键创建聚簇索引（Clustered Index），InnoDB表的数据行物理上按照主键顺序存储。因此按`id`查询（`WHERE id = ?`）速度最快
3. **非空保证**：主键列自动包含`NOT NULL`约束

**为什么选`id`作为主键**：
- 自增整数主键是InnoDB的最佳实践。自增保证新数据总是追加到B+树的末尾，减少页分裂
- 相比用`username`做主键，整数主键占空间更小，索引效率更高，且用户名可能变更

---

```sql
    UNIQUE KEY uk_username (username)
```

**`UNIQUE KEY uk_username (username)`** — 唯一索引：

**逐部分解析**：
- `UNIQUE KEY`：唯一索引。与普通索引的区别是索引列的值必须唯一（不允许重复）
- `uk_username`：索引名称。命名规范：`uk`前缀表示Unique Key，`username`表示索引列名
- `(username)`：索引包含的列

**唯一索引的作用**：
1. **数据完整性**：数据库层面阻止重复用户名的插入。如果两条INSERT语句试图插入相同的`username`，第二条会报错`Duplicate entry 'xxx' for key 'uk_username'`
2. **查询加速**：`WHERE username = ?`查询可以利用此索引快速定位，避免全表扫描
3. **双重保险**：应用层（Controller）在插入前检查用户名是否存在，数据库层通过唯一索引提供最后的防线

```sql
    username  VARCHAR(50)  NOT NULL DEFAULT ''     COMMENT '上传者用户名',
```

**`username`字段解析**：

| 属性 | 值 | 含义 |
|------|-----|------|
| 数据类型 | `VARCHAR(50)` | 与`user.username`类型一致 |
| `NOT NULL` | — | 非空 |
| `DEFAULT ''` | — | 默认值为空字符串 |

**设计意图**：这是一个**冗余字段**——username信息也可以通过`user_id` JOIN user表获取。但在此直接存储username有以下好处：
1. **查询性能**：`SELECT * FROM resource ORDER BY id DESC`无需JOIN即可直接获取上传者用户名，避免每次加载资源列表都做表连接
2. **数据稳定**：即使用户后续修改了用户名，资源记录仍保留上传时的原始用户名
3. **查询简化**：前端渲染资源卡片直接使用`item.username`，后端SQL不需要LEFT JOIN

**冗余设计权衡**：
- 优点：查询快，SQL简单
- 缺点：数据可能不一致（用户改名后此字段不会自动更新），占用额外存储空间
- 在本项目中，用户无法修改用户名，且数据量小，冗余存储是合理的设计选择

---

```sql
    PRIMARY KEY (id),
    KEY idx_user_id (user_id)
```

**索引设计**：

`PRIMARY KEY (id)` — 主键，与user表相同。

`KEY idx_user_id (user_id)` — **普通索引**（非唯一索引）：
- `KEY`：MySQL中`KEY`和`INDEX`是同义词
- `idx_user_id`：索引名称，`idx`前缀表示普通索引
- `(user_id)`：索引列

**索引用途**：加速按用户ID查询资源的操作。如果应用有"查看某用户上传的所有资源"功能，`WHERE user_id = ?`可以利用此索引快速定位。虽然当前前端没有此功能，但该索引为未来扩展提供了基础。

### 2.4 评论表（comment）建表语句

```sql
-- ============================================================
-- 3. 评论表（5 个字段）
-- ============================================================
CREATE TABLE comment (
    id          INT          NOT NULL AUTO_INCREMENT COMMENT '评论ID',
```

**`id`字段**：评论唯一标识，INT自增主键。在API中用于删除评论时的路径参数（`/api/resources/{rId}/comments/{cId}`中的`cId`）。

---

```sql
    resource_id INT          NOT NULL                COMMENT '资源ID',
```

**`resource_id`字段解析**：

| 属性 | 值 | 含义 |
|------|-----|------|
| 数据类型 | `INT` | 与`resource.id`类型一致 |
| `NOT NULL` | — | 必填，每条评论必须关联一个资源 |

**业务含义**：建立comment表与resource表的关联。
- 评论必须依附于某个资源，`resource_id`标识了被评论的目标资源
- 查询某资源的所有评论：`WHERE resource_id = ?`（最频繁的查询，有索引加速）
- 如果资源被删除，其关联评论的处理逻辑由应用层决定（当前项目未处理此场景）

---

```sql
    user_id     INT          NOT NULL                COMMENT '评论者ID',
```

**`user_id`字段**：

| 属性 | 值 | 含义 |
|------|-----|------|
| 数据类型 | `INT` | 与`user.id`类型一致 |
| `NOT NULL` | — | 必填，每条评论必须有明确的发表者 |

**业务含义**：
- 用于权限控制：删除评论时比对`user_id`是否等于当前登录用户ID
- 用于JOIN查询获取评论者用户名（`LEFT JOIN user u ON c.user_id = u.id`）

---

```sql
    username    VARCHAR(50)  NOT NULL DEFAULT ''     COMMENT '评论者用户名',
```

**`username`字段**：冗余字段，与resource表中的username设计思路一致。但在评论表中，INSERT语句**不包含此字段**（`INSERT INTO comment (resource_id, user_id, content) VALUES ...`），而是通过SELECT时的LEFT JOIN动态获取用户名。因此评论表的username字段利用了DEFAULT值`''`，实际查询返回的用户名来自user表。

---

```sql
    content     TEXT         NOT NULL                COMMENT '评论内容',
```

**`content`字段解析**：

| 属性 | 值 | 含义 |
|------|-----|------|
| 数据类型 | `TEXT` | 文本类型，最大存储65,535字节（约64KB）。相比VARCHAR，TEXT不能设默认值且存储方式不同（数据存储在独立的区域） |
| `NOT NULL` | — | 必填 |

**为什么用TEXT而非VARCHAR**：
- `VARCHAR`最大65,535字节（MySQL行大小限制），TEXT同样最大65,535字节但不受行大小严格限制
- 评论内容可能较长（虽然前端限制500字符，但数据库设计为TEXT提供了扩展空间）
- `TEXT`类型不能设置DEFAULT值（MySQL限制），但因为设置了NOT NULL，插入时必须提供值

---

```sql
    PRIMARY KEY (id),
    KEY idx_resource_id (resource_id),
    KEY idx_user_id (user_id)
```

**三个索引**：

| 索引名 | 类型 | 列 | 用途 |
|--------|------|-----|------|
| `PRIMARY KEY` | 主键/聚簇 | `id` | 主键索引，加速按评论ID的精确查询 |
| `idx_resource_id` | 普通索引 | `resource_id` | 加速按资源ID查询评论列表（`WHERE resource_id = ?`），这是最频繁的查询 |
| `idx_user_id` | 普通索引 | `user_id` | 加速按用户ID查询评论（如"我的评论"功能），为未来扩展预留 |

**索引设计考量**：
- `idx_resource_id`是最重要的性能索引——每次打开资源详情都要查询该资源的所有评论
- 在`resource_id`上建索引可以将查询时间复杂度从O(n)的全表扫描降为O(log n)的B+树查找

## 三、所有SQL语句逐行解析

项目中，MyBatis Mapper XML文件共定义了**9条SQL语句**，覆盖全部业务功能的增删改查操作。

### 3.1 用户相关SQL（2条）

#### SQL-1：插入新用户（注册）

**文件位置**：`UserMapper.xml` → `<insert id="insert">`

```sql
INSERT INTO user (username, password) VALUES (#{username}, #{password})
```

**逐部分解析**：

`INSERT INTO user (username, password)` — 向user表插入一条新记录，显式指定了插入列（`id`列未指定，由AUTO_INCREMENT自动生成）。

`VALUES (#{username}, #{password})` — 值列表：
- `#{username}`：MyBatis参数占位符，运行时替换为`User`对象的`username`属性值（如`'zhangsan'`）
- `#{password}`：运行时替换为`password`属性值（如`'123456'`）
- MyBatis使用`PreparedStatement`（预处理语句），实际执行的SQL为`INSERT INTO user (username, password) VALUES (?, ?)`，参数值通过`?`占位符安全传入，防止SQL注入

**调用时机**：用户注册时（`UserController.register()`）。

**执行效果示例**：

| 执行前 | user表数据 | 执行后 |
|--------|-----------|--------|
| id=1, username='zhangsan' | → INSERT → | id=1, username='zhangsan' |
| | | id=2, username='lisi'（新增行） |

**MyBatis配置补充**：`useGeneratedKeys="true" keyProperty="id"`使INSERT执行后，数据库自动生成的id值回写到传入的User对象中。即执行前`user.getId()=null`，执行后`user.getId()=2`。

---

#### SQL-2：按用户名查询用户（登录/去重校验）

**文件位置**：`UserMapper.xml` → `<select id="selectByUsername">`

```sql
SELECT id, username, password FROM user WHERE username = #{username}
```

**逐部分解析**：

`SELECT id, username, password` — 查询三列：
- `id`：用户唯一标识
- `username`：用户名
- `password`：密码（用于登录时比对）

`FROM user` — 从user表中查询。

`WHERE username = #{username}` — 按用户名精确匹配。利用`uk_username`唯一索引加速查询。因为username有唯一约束，查询结果要么0行（用户不存在），要么1行（命中）。

**调用时机**：
1. 注册时：检查用户名是否已存在（`UserController.register()`中`selectByUsername() != null`判断）
2. 登录时：获取用户信息进行密码验证（`AuthController.login()`）

**执行效果**：
```sql
-- 输入参数：#{username} = 'zhangsan'
SELECT id, username, password FROM user WHERE username = 'zhangsan'
-- 返回：id=1, username='zhangsan', password='123456'
```

#### SQL-4：查询全部资源（资源广场列表）

**文件位置**：`ResourceMapper.xml` → `<select id="selectAll">`

```sql
SELECT id, title, file_path, file_name, user_id, username
FROM resource ORDER BY id DESC
```

**逐部分解析**：

查询所有6个字段：id（资源ID）、title（标题）、file_path（存储路径）、file_name（原始文件名）、user_id（上传者ID）、username（上传者用户名）。

`FROM resource` — 从resource表查询，无WHERE条件表示查询所有记录。

`ORDER BY id DESC` — 按资源ID**降序**排列：
- `DESC`：Descending，降序。ID大的（最新上传的）排在最前面
- 如果使用`ASC`（升序），最早上传的资源排在最前面
- 资源广场展示时，用户期望看到最新资源，所以降序排列

**调用时机**：进入资源广场页面时（`ResourceController.list()`），以及删除资源后刷新列表时。

**性能说明**：此查询无分页（LIMIT），在资源量大时可能造成性能问题。当前适用于学习平台的初期阶段（资源量预计在百/千级别）。

#### SQL-5：按ID查询单条资源（下载/删除前的查询）

**文件位置**：`ResourceMapper.xml` → `<select id="selectById">`

```sql
SELECT id, title, file_path, file_name, user_id, username
FROM resource WHERE id = #{id}
```

**`WHERE id = #{id}`** — 按主键精确查询：
- 利用主键索引（聚簇索引），查询效率最高（O(log n) B+树查找）
- 只返回0行或1行（主键唯一）
- 参数`#{id}`绑定自方法参数`Integer id`

**调用时机**：
1. 下载资源时：获取文件路径和原始文件名
2. 删除资源时：获取资源记录进行存在性检查和权限校验

---

#### SQL-6：按ID删除资源

**文件位置**：`ResourceMapper.xml` → `<delete id="deleteById">`

```sql
DELETE FROM resource WHERE id = #{id}
```

**`DELETE FROM resource`** — 从resource表中删除记录。

**`WHERE id = #{id}`** — 按主键删除，确保只删除指定的唯一一条记录。参数通过`@Param("id")`绑定。

**调用时机**：用户点击删除按钮并通过权限校验后（`ResourceController.delete()`）。

**注意事项**：
1. 此SQL只删除数据库记录，**不删除服务器磁盘上的文件**。长期运行后`uploads/`目录中可能存在"孤儿文件"（数据库记录已删除但文件仍存在）
2. 删除操作不可逆（没有软删除/回收站机制），用户确认对话框是唯一的安全网
3. 如果该资源有评论，评论记录不会级联删除（数据库中未设置外键级联），将形成"孤儿评论"

#### SQL-7：插入新评论

**文件位置**：`CommentMapper.xml` → `<insert id="insert">`

```sql
INSERT INTO comment (resource_id, user_id, content)
```

**逐字段绑定解析**：

| SQL参数 | Java属性 | 示例值 | 说明 |

| `#{content}` | `comment.content` | `'谢谢分享！'` | 评论正文内容 |

**调用时机**：用户在资源详情面板中输入评论内容并点击"发送"按钮后（`CommentController.create()`）。

#### SQL-8：按资源ID查询评论列表（含JOIN用户名）

**文件位置**：`CommentMapper.xml` → `<select id="selectByResourceId">`

```sql
SELECT c.id, c.resource_id, c.user_id, c.content, u.username
FROM comment c LEFT JOIN user u ON c.user_id = u.id
WHERE c.resource_id = #{resourceId} ORDER BY c.id DESC
```

**逐子句解析**：

**① SELECT子句**：
```sql
SELECT c.id, c.resource_id, c.user_id, c.content, u.username
```
- `c.id`、`c.resource_id`、`c.user_id`、`c.content`：来自comment表（别名`c`）
- `u.username`：来自user表（别名`u`），通过JOIN获取评论者的当前用户名
- 查询结果被MyBatis自动映射到`Comment`对象（`username`字段来自`u.username`而非`c.username`）

**② FROM子句 — LEFT JOIN**：
```sql
FROM comment c LEFT JOIN user u ON c.user_id = u.id
```
- `comment c`：comment表设置别名`c`
- `LEFT JOIN user u`：左外连接user表，别名`u`
- `ON c.user_id = u.id`：连接条件——评论的user_id等于用户的id
- **LEFT JOIN的含义**：以comment（左表）为主，即使右表（user）中没有匹配的记录，左表数据仍会返回。例如：极端情况下用户被删除了，评论仍然显示，只是username为NULL

**③ WHERE子句**：
```sql
WHERE c.resource_id = #{resourceId}
```
- 筛选特定资源的所有评论
- 利用`idx_resource_id`索引加速查询

**④ ORDER BY子句**：
```sql
ORDER BY c.id DESC
```
- 按评论ID降序，最新评论（ID大）排在最前面
- 等同于按发表时间倒序（因为ID自增，ID大 = 后发表）

**调用时机**：用户在资源广场页面点击资源打开详情面板时（`CommentController.list()`），以及发表/删除评论后刷新评论列表时。

---

#### SQL-9：按ID查询单条评论（删除权限校验）

**文件位置**：`CommentMapper.xml` → `<select id="selectById">`

```sql
SELECT id, resource_id, user_id, content FROM comment WHERE id = #{id}
```

**与SQL-8的差异**：
- 不包含`u.username`和LEFT JOIN：因为此查询用于权限校验，只需要比较`user_id`，不需要展示用户名
- 只查询4个字段（不包括username）

**调用时机**：用户删除评论前（`CommentController.delete()`），用于检查评论是否存在以及当前用户是否有权删除。

### 4.3 SQL执行频率与性能分析

| SQL编号 | 业务功能 | 执行频率 | 查询方式 | 索引利用 | 性能评估 |
|---------|---------|---------|---------|---------|---------|
| SQL-1 | 用户注册 | 低（新用户注册时） | INSERT | — | 快（单行插入） |
| SQL-2 | 登录/去重检查 | 中（登录+注册时） | WHERE username=? | uk_username唯一索引 | 极快（唯一索引精确查找） |
| SQL-3 | 上传资源 | 中（用户上传时） | INSERT | — | 快（单行插入） |
| SQL-4 | 资源列表 | 高（每次进入资源广场） | 全表+ORDER BY | PRIMARY KEY（排序用） | 中等（全表扫描+排序，无分页） |
| SQL-5 | 下载/删除前查询 | 中（下载+删除时） | WHERE id=? | PRIMARY KEY | 极快（主键聚簇索引查找） |
| SQL-6 | 删除资源 | 低（资源所有者操作） | WHERE id=? | PRIMARY KEY | 极快（主键删除） |
| SQL-7 | 发表评论 | 中（用户评论时） | INSERT | — | 快（单行插入） |
| SQL-8 | 评论列表 | 高（打开资源详情时） | WHERE resource_id=? + JOIN + ORDER BY | idx_resource_id | 快（索引查找+JOIN+排序） |
| SQL-9 | 删除评论前查询 | 低（评论者操作） | WHERE id=? | PRIMARY KEY | 极快（主键查找） |
| SQL-10 | 删除评论 | 低 | WHERE id=? | PRIMARY KEY | 极快（主键删除） |
